Code is Cheap

The problem of software engineering has changed

Today we can write code in minutes with artificial intelligence. Code is no longer the bottleneck.

What matters now:

  • Planning — what to build and why
  • Reviewing — ensuring what was built is correct
  • Ensuring quality — tests, security, observability

An LLM is a language model, not a code model. It predicts the next token. If you don't provide context, it predicts based on what it learned from thousands of texts — probably not what you want.

"Don't delegate responsibility. Delegate execution."

The responsibility of defining what the AI will write, of reviewing, of approving — is all yours.

What is Claude Code?

An agentic AI tool in the terminal

  • Terminal-first — runs where you work, no context switching
  • Available on: CLI, VS Code, JetBrains, web, desktop, Slack
  • Reads the codebase, edits files, runs commands, manages git
  • Continuous loop: read → think → act → verify

Setting Up the Environment

CLAUDE.md — The file that changes everything

Local CLAUDE.md

Your personal preferences:

  • How you like to code
  • Functional programming, DDD, etc.
  • Available skills
  • Security rules
  • Communication tone for PRs

Project CLAUDE.md

Team conventions:

  • Project architecture
  • Commands (make test, make lint)
  • Development workflow
  • PR checklist
  • Testing patterns

The agent reads these files automatically when starting each session. Without them, the agent has no context — and will guess.

You can have CLAUDE.md in each module of the project. Large projects use subfolders like .claude/code-reviewer/reference/.

# TDD Skill

## Workflow
1. Read the plan/spec for the task
2. Write test cases FIRST (they must fail)
3. Implement code to make tests pass
4. Run tests to verify
5. Refactor if needed, keeping tests green

## Rules
- Never skip the failing test step
- Use the project's test framework
- One test file per feature/module

Spec Driven Development

Define the spec. The agent executes.

You don't ask the AI to "code something". You define exactly what you want:

  1. Ticket — what's the problem, what's the context (Linear, Jira, GitHub Issue)
  2. Technical context — how the system works, which patterns to follow
  3. Rules — what it can and cannot do
  4. Expected result — PRs opened, tests passing, docs updated
You: "This issue is about adding a new slide on MCP.
      The project is pure HTML with Catppuccin Macchiato theme.
      Follow CLAUDE.md and keep PT-BR and EN in sync."

Claude: [reads CLAUDE.md, AGENTS.md, update-slides skill]
        [analyzes the structure of all 33 existing slides]
        [proposes a detailed plan]

The agent explores, reads, understands — and proposes a plan. You don't accept it blindly. You review the plan.

PRD — Product Requirements Document

Before coding, design

Use AI to co-create the PRD for what you're going to build:

  • Describe what the product/feature does
  • Technical decisions: stack, infra, authentication, database
  • Endpoints and contracts (Zod, TypeScript strict)
  • Milestones with checkboxes — the agent checks them off as it progresses
## PRD: DialogFlow Client Refactoring

### Technical decisions
- Kotlin coroutines for non-blocking calls
- Feature flag to control rollback
- OpenTelemetry for instrumentation
- Deprecate old classes (don't remove immediately)

### Milestones
- [ ] M0: Create interfaces and feature flag
- [ ] M1: Implement non-blocking client
- [x] M2: Unit and integration tests
- [ ] M3: Migrate existing calls
"Programming starts with engineering. Every decision made in the PRD saves 4-5 bug fixes down the road."

In Practice: PRD + Spec

The contract between you and the agent

PRD — What to build

# PRD: Notification System

## Goal
Users receive real-time notifications
about project events.

## Technical decisions
- WebSocket with SSE fallback
- PostgreSQL + Redis pub/sub
- Feature flag: notifications_v2

## Milestones
- [ ] M0: Schema + migration
- [ ] M1: WebSocket server
- [x] M2: E2E tests
- [ ] M3: Preferences UI

Spec — How to verify

# Spec: WebSocket Notification Server

## Goal
Real-time event delivery to clients.

## Acceptance Criteria
- GIVEN user connected via WebSocket
  WHEN an event triggers
  THEN notification arrives in < 500ms

## Non-goals
- Push notifications (mobile)
- Email digest

## Edge Cases
- Disconnect → retry queue
- Redis down → fallback to SSE
"The PRD defines WHAT to build. The Spec defines HOW to verify it works. Together, they are the contract between you and the agent."

The Plan is Everything

Approve before coding. Always.

The most important point of the workflow: give the agent autonomy ONLY after approving the plan.

         ┌──────────────────┐
         │  You define       │
         │  the problem      │
         └────────┬─────────┘
                  ▼
         ┌──────────────────┐
         │ Agent explores   │◄──── Reject + refine
         │ and proposes plan│      (feature flags,

         ┌──────────────────┐

                  ▼
         ┌──────────────────┐

         └──────────────────┘

Don't accept the first draft. Iterate. Refine. Add constraints.

TDD with AI

Tests first. Make them fail. Implement. Iterate.

The agent writes tests before the implementation:

  1. Writes test scenarios based on the approved plan
  2. Tests fail (classes don't exist yet)
  3. Implements the code to make the tests pass
  4. Runs tests again — if they fail, fixes and repeats

The secret: give the agent tools to verify whether its work is correct.

Without verificationWith verification
Agent says it's doneAgent runs tests, linter, build
You open the PRIf it fails, the agent fixes it on its own
Doesn't compileDelivers a PR that compiles and passes
Endless back-and-forthAutonomous loop until it works
"If the agent doesn't have tools to know whether its work is correct, you'll waste time going back and forth."

PR Reviews with AI

You review first. The agent complements.

Use plan mode for review — the agent analyzes but does nothing.

                ┌─ Code Quality ──── architecture, DDD, patterns
Colleague's PR ►├─ Performance ───── blocking calls, N+1
                ├─ Test Coverage ─── missing scenarios
                └─ Security ──────── PII, injection, dependencies
                          │
                Consolidation + Comments via API

4 subagents run in parallel, each one specialized.

But beware:

  • You review first, without Claude's context, to maintain your judgment
  • The agent complements your review — it doesn't replace it
  • If you start delegating everything, you'll develop the habit of accepting everything
  • Reviewing code is your responsibility

Responding to Reviews

Read, evaluate, resolve

When people review your PRs:

  1. You read the comments and evaluate which ones make sense
  2. Have the agent resolve + respond to the ones you agree with
  3. The agent makes the fix, commits, and replies to the comment
$ claude "Read the comments on PR #42.
  See which ones make sense and resolve them.
  Reply to the comments explaining what changed."

Security and Guardrails

Control what the agent can do

Hooks

Code that runs at lifecycle points:

  • PreToolUse — block rm -rf, force push
  • PostToolUse — run linter, formatter
  • Stop — notify when done
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",

      }]

  }
}

Rules in CLAUDE.md

Explicitly state what it cannot do:

  • Don't touch production data
  • Don't modify main/master directly
  • Don't make changes to third-party APIs
  • Don't commit secrets or credentials

Human-on-the-Loop

  • Every generated PR requires human review
  • Fixes go through tests first
  • Never silent mutation

Working in Parallel

Multiple agents on the same codebase

Each terminal session is an independent agent. The problem: two agents editing the same file at the same time creates conflicts.

Git Worktrees — the solution

A worktree is a working copy of your repo in another directory, sharing the same .git. Each worktree is on a different branch, without cloning anything. It's instant.

# Create a worktree for each task (instant, no clone)
git worktree add ../feature-auth feature/auth
git worktree add ../feature-api feature/api

# List active worktrees
$ git worktree list
/home/user/project           abc1234 [main]
/home/user/feature-auth      def5678 [feature/auth]
/home/user/feature-api       ghi9012 [feature/api]

# Remove when done
git worktree remove ../feature-auth

Worktree skill: a markdown that teaches the agent to create worktrees automatically. If on main, it creates a new worktree before starting.

Tab 1Tab 2Tab 3
Implementing feature AReviewing colleague's PRCreating tickets in Linear
../feature-a/plan mode (read-only)MCP Linear

Set up the terminal to notify when each agent finishes. You can grab a coffee and come back when the notification arrives.

Next Steps

Where to start

  1. Create your CLAUDE.md — 10 minutes that change every future session
  2. Contextualize your repos — architecture.md + CONTRIBUTING.md
  3. Adopt Spec Driven Development — define the problem before coding
  4. Create a PRD before large features — milestones with checkboxes
  5. Use plan mode — approve the plan before giving autonomy
  6. Give verification tools — tests, linter, build
  7. Install hooks for automatic guardrails
  8. Then — and only then — scale to automation with CI/CD
"The order matters: first the individual flow, then the team flow. Automating a bad process only scales the problems."
