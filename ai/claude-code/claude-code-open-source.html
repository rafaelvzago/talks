Código é Barato

O problema da engenharia de software mudou

Hoje a gente consegue escrever código em minutos com inteligência artificial. Código não é mais o gargalo.

O que importa agora:

  • Planejar — o que construir e por que
  • Revisar — garantir que o que foi construído está correto
  • Garantir qualidade — testes, segurança, observabilidade

LLM é modelo de linguagem, não modelo de código. Ela prevê o próximo token. Se você não dá contexto, ela prevê o que aprendeu com milhares de textos — provavelmente não é o que você quer.

"Não delegue responsabilidade. Delegue execução."

A responsabilidade de definir o que a IA vai escrever, de revisar, de aprovar — é toda sua.

Um formato aberto usado por 60K+ projetos open source

# AGENTS.md

## Comandos de setup
- Instalar deps: `pnpm install`
- Dev server: `pnpm dev`
- Rodar testes: `pnpm test`

## Estilo de código
- TypeScript strict mode
- Aspas simples, sem ponto e vírgula
- Preferir padrões funcionais

Por que AGENTS.md?

  • Suporta aninhamento em monorepos — cada subprojeto com seu AGENTS.md

  • Mantido pela Linux Foundation (Agentic AI Foundation)

Exemplo de Skill — skill-tdd.md

Instruções que o agente carrega automaticamente

---
name: tdd
description: >
  Use when writing tests first, doing TDD,
  or when the user says "write tests" or
  "test-driven". Writes failing tests,
  then implements code to pass them.
---

# TDD Skill

## Workflow
1. Read the plan/spec for the task
2. Write test cases FIRST (they must fail)
3. Implement code to make tests pass
4. Run tests to verify
5. Refactor if needed, keeping tests green

## Rules
- Never skip the failing test step
- Use the project's test framework
- One test file per feature/module

Como funciona

  • Cada skill é uma pasta com SKILL.md
  • O campo description define quando o agente ativa a skill automaticamente
  • Skills podem conter scripts, templates e referências auxiliares
  • O agente lê a skill e segue as instruções como um especialista

Spec Driven Development

Defina a spec. O agente executa.

Você não pede para a IA "codar algo". Você define exatamente o que quer:

  1. Ticket — qual o problema, qual o contexto (Linear, Jira, GitHub Issue)
  2. Contexto técnico — como o sistema funciona, quais padrões seguir
  3. Regras — o que pode e o que não pode fazer
  4. Resultado esperado — PRs abertos, testes passando, docs atualizados
Você: "Essa issue é sobre adicionar um slide novo sobre MCP.
       O projeto é HTML puro com tema Catppuccin Macchiato.
       Siga o CLAUDE.md e mantenha PT-BR e EN sincronizados."

Claude: [lê CLAUDE.md, AGENTS.md, skill update-slides]
        [analisa a estrutura dos 33 slides existentes]
        [propõe um plano detalhado]

O agente explora, le, entende — e propõe um plano. Você não aceita cegamente. Você revisa o plano.

PRD — Product Requirements Document

Antes de codar, projete

Use IA para co-criar o PRD do que você vai construir:

  • Descrever o que o produto/feature faz
  • Decisões técnicas: stack, infra, autenticação, banco de dados
  • Endpoints e contratos (Zod, TypeScript strict)
  • Milestones com checkboxes — o agente vai ticando conforme avanca
## PRD: Refatoração do Cliente DialogFlow

### Decisões técnicas
- Kotlin coroutines para chamadas não-blocantes
- Feature flag para controlar rollback
- OpenTelemetry para instrumentação
- Deprecar classes antigas (não remover imediatamente)

### Milestones
- [ ] M0: Criar interfaces e feature flag
- [ ] M1: Implementar cliente não-blocante
- [x] M2: Testes unitários e de integração
- [ ] M3: Migrar chamadas existentes
"Programação comeca na engenharia. Cada decisao tomada no PRD economiza 4-5 bug fixes la na frente."

Na Prática: PRD + Spec

O contrato entre você e o agente

PRD — O que construir

# PRD: Sistema de Notificações

## Objetivo
Usuários recebem notificações em tempo
real sobre eventos do projeto.

## Decisões técnicas
- WebSocket com fallback para SSE
- PostgreSQL + Redis pub/sub
- Feature flag: notifications_v2

## Milestones
- [ ] M0: Schema + migration
- [ ] M1: WebSocket server
- [x] M2: Testes e2e
- [ ] M3: UI de preferências

Spec — Como verificar

# Spec: WebSocket Notification Server

## Goal
Real-time event delivery to clients.

## Acceptance Criteria
- GIVEN user connected via WebSocket
  WHEN an event triggers
  THEN notification arrives in < 500ms

## Non-goals
- Push notifications (mobile)
- Email digest

## Edge Cases
- Disconnect → retry queue
- Redis down → fallback to SSE
"O PRD define O QUE construir. A Spec define COMO verificar que funciona. Juntos, eles são o contrato entre você e o agente."

O Plano é Tudo

Aprovar antes de codar. Sempre.

O ponto mais importante do workflow: dar autonomia ao agente SÓ depois de aprovar o plano.

Não aceite o primeiro rascunho. Itere. Refine. Adicione restrições.

TDD com IA

Testes primeiro. Fazer falhar. Implementar. Iterar.

O agente escreve testes antes da implementação:

  1. Escreve cenários de teste baseados no plano aprovado
  2. Testes falham (classes ainda não existem)
  3. Implementa o código para fazer os testes passarem
  4. Roda testes novamente — se falham, corrige e repete

O segredo: dar ao agente ferramentas para verificar se o trabalho dele está certo.

Sem verificaçãoCom verificação
Agente diz que terminouAgente roda testes, linter, build
Você sobe o PRSe falha, o agente corrige sozinho
Não compilaEntrega PR que compila e passa
Vai-e-volta eternoLoop autônomo até funcionar
"Se o agente não tem ferramentas para saber se o trabalho está certo, você vai perder tempo no vai-e-volta."

Review de PRs com IA

Você revisa primeiro. O agente complementa.

Use plan mode para review — o agente analisa mas não faz nada.

                ┌─ Code Quality ──── arquitetura, DDD, padrões
PR do colega ──►├─ Performance ───── chamadas blocantes, N+1
                ├─ Test Coverage ─── cenários faltando
                └─ Security ──────── PII, injeção, dependências
                          │
                Consolidação + Comentários via API

4 subagentes rodam em paralelo, cada um especializado.

Mas atenção:

  • Você revisa primeiro, sem contexto do Claude, para manter a sanidade
  • O agente complementa sua revisão — não substitui
  • Se você começar a delegar tudo, vai criar o hábito de aceitar tudo
  • Revisar código é responsabilidade sua

Respondendo Reviews

Ler, avaliar, resolver

Quando revisam seus PRs:

  1. Você le os comentários e avalia quais fazem sentido
  2. Manda o agente resolver + responder os que você concorda
  3. O agente faz o fix, commita, e responde o comentario

Trabalhando em Paralelo

Múltiplos agentes no mesmo codebase

Cada sessão do terminal é um agente independente. O problema: dois agentes editando o mesmo arquivo ao mesmo tempo gera conflito.

Git Worktrees — a solução

Uma worktree é uma cópia de trabalho do seu repo em outro diretório, compartilhando o mesmo .git. Cada worktree fica em uma branch diferente, sem clonar nada. É instantâneo.

# Criar uma worktree para cada tarefa (instantâneo, sem clone)
git worktree add ../feature-auth feature/auth
git worktree add ../feature-api feature/api

# Listar worktrees ativas
$ git worktree list
/home/user/projeto          abc1234 [main]
/home/user/feature-auth     def5678 [feature/auth]
/home/user/feature-api      ghi9012 [feature/api]

# Remover quando terminar
git worktree remove ../feature-auth

Skill para worktrees: um markdown que ensina o agente a criar worktrees automaticamente. Se está na main, cria uma worktree nova antes de começar.

Tab 1Tab 2Tab 3
Implementando feature ARevisando PR do colegaCriando tickets no Linear
../feature-a/plan mode (read-only)MCP Linear

Configure o terminal para notificar quando cada agente terminar. Você pode tomar café e voltar quando a notificação chegar.

Próximos Passos

Por onde começar

  1. Crie seu CLAUDE.md — 10 minutos que mudam toda sessão futura
  2. Contextualize seus repos — architecture.md + CONTRIBUTING.md
  3. Adote Spec Driven Development — defina o problema antes de codar
  4. Crie um PRD antes de features grandes — milestones com checkboxes
  5. Use plan mode — aprove o plano antes de dar autonomia
  6. De ferramentas de verificação — testes, linter, build
  7. Instale hooks para guardrails automáticos
  8. Depois — e só depois — escale para automação com CI/CD
"A ordem importa: primeiro o fluxo individual, depois o fluxo do time. Automatizar um processo ruim só escala os problemas."
